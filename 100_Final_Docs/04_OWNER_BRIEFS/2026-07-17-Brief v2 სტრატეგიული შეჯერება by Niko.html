4. Codex-ის დამოუკიდებელი კრიტიკა (07-17, verbatim ჩაწერილია CHALLENGE §2-ში)

Codex-მა ორივე Stage-1 დოკუმენტი თავად წაიკითხა და დავალებისამებრ უარყოფაზე იმუშავა. 5 შეტევა:

  1. Build-shape: „აუდიტმა ბაზარი კი არ იპოვა, შიდა პრეფერენცია აქცია გადაწყვეტილებად." 0 ინტერვიუ, 0 გადახდა, 1 დაბრუნებული მოწყობილობა; ყველა არგუმენტი supply-side-ია და არა მოთხოვნის მტკიცებულება. „3-7" ერთი სეგმენტიც არაა: 3 წლის ფონოლოგია და 7 წლის დეკოდირება სხვადასხვა პროდუქტია.
  2. D2/D3: hybrid privacy = consent theater, სანამ თანხმობის არქიტექტურა (ვინ თანხმდება, რა ინახება, როგორ იშლება) განუსაზღვრელია; opt-in კოჰორტა selection bias-ით თვით outcome-აქტივს აბინძურებს. D3-ის 8-წლის ზღვარი დაუსაბუთებელი დოქტრინაა; არსებობს ტესტირებადი შუალედი: runtime მოდელი დეტერმინისტული state machine-ის უკან, შემოსაზღვრული ფუნქციებით.
  3. Moat: „proposed database is not a moat." რიცხვები: efficacy-განაცხადი ≈ 200-400 დამასრულებელი; გამოსადეგი dataset ≈ 1,000; დაცვადობა ≈ 3,000-5,000 ლონგიტუდინალური მოსწავლე. დისტრიბუცია flywheel-ის წინაპირობაა და არა „არხი". უფრო ძლიერი აქტივები: ვალიდირებული ქართული შეფასების მეთოდი, პედაგოგ-ურთიერთობები, კონტენტის უფლებები, დისტრიბუცია.
  4. ექსპერიმენტები: E1 ცვლადებს ისე კრებს, რომ მიზეზს ვერ გამოყოფ; თბილი კოჰორტა + weekly check-in = უკვე სერვისი; E1-ის წარმატება coached-service-ს დაადასტურებს და არა pure-AI პროდუქტს. E2-ის „≥3 გადამხდელი" არასტაბილურია; ქცევითი endpoint ფულია და არა ინტერესი.
  5. რეკომენდაცია: standalone პროდუქტზე CONDITIONAL-GO უარყავი. ჯერ ფასიანი, პედაგოგ-წაყვანილი human+AI სერვისი (კვალიფიციური ქართული ლიტერაციის პედაგოგი revenue share-ზე, founder ≤1სთ/დღე ოპერაციებზე), აპი = მიწოდების და გაზომვის შიდა ინსტრუმენტი. თუ პედაგოგი და წინასწარ გადამხდელი ოჯახები ვერ მოიძებნა, გაჩერდი.
