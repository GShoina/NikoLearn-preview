## Why this file exists

The audit self-declares **static-source-only** (`runtime NOT VERIFIABLE` appears in its own text). So every
one of its 29 findings arrives as a **HYPOTHESIS**, not a fact. Two failure modes are equally expensive:
fixing a false positive (wasted work + a change nobody needed), and dismissing a real one (a child or
parent gets hurt by something we were told about and shrugged off). Both are prevented the same way: a
per-finding verdict with a **named check method**, written down permanently.

**A disproved finding stays in this table with its disproof** (ჩანაცვლება ≠ წაშლა). Deleting it would
mean the same suspicion returns from zero the next time anyone reads the audit — and the cost of
re-verifying is paid again. A FALSE verdict is a durable asset, not a scrap.

**Absence ≠ clean** (§14): a row claiming "no instance" must name the concrete check that looked. This
already caught us twice on this audit's own findings (NB-52's missing regex `g` flag reported a false
"NONE"; NB-53's first test read `window.gate`, which `let`-globals never populate — a test looking at the
wrong place proves nothing).

## Verdict vocabulary

| Verdict | Meaning |
|---|---|
| **CONFIRMED** | The code claim is true, verified against the real file. |
| **FALSE** | The claim is wrong. The disproof is recorded and this finding is closed. |
| **PARTIAL** | One half true, one half not. The row says which. |
| **STALE** | Was true, already fixed. Names the fix. |
| **NOT-STATICALLY-VERIFIABLE** | Needs the app running. Names the exact runtime check owed. |
| **PENDING** | Not yet checked. Not a verdict — an admission. |

Severity is the AUDIT's number → OURS. We re-rate freely: an audit rates the code, we rate the harm to a
child or a parent. `NB-52` is the worked example (Sev-3 → Sev-2, because blast radius was business
strategy with **zero child PII** — scoping the harm, not softening it).
