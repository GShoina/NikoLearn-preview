# VOICE CREATION GATE — NikoLearn audio-identity standard

> **Status:** ACTIVE standard · created 2026-07-22 (PI-104, triggered by NB-89) · owner: Gela Shonia
> **What it is:** the audio sibling of the §6c visual-verification gate. Any audio created OUTSIDE
> the running app (deliverable narration, video-tale voice, future recorded clips) MUST pass this
> gate before it ships, so every voice a child or parent hears is the SAME NikoLearn voice identity.
> **Trigger:** NB-89 — the „1 apple" deliverable's English voice sounded different from the app's own
> English voice (tempo, clarity, tonality). Root cause below.

**Relationship to the other two voice docs (one axis each, no overlap — the trio):**
- `VOICE_STANDARD.md` = the **character** recipe (warm Georgian teacher, ღ/ყ/ძ pronunciation gate).
- `VOICE_QUALITY_STANDARD.md` (+ `tools/voice-eval.mjs`) = **is the clip good?** (physics + WER + proxy-ear).
- **THIS doc** = **does the clip match our ONE identity?** the consistency/creation gate every non-runtime
  clip passes. A clip can be high quality (passes voice-eval) yet still be the WRONG narrator (NB-89) — that
  is the exact gap this closes. Use all three: character → quality → identity.

---

## 1. The canonical voice profile (the ONE identity) — [FACT, from `niko/core.js`]

| Language | How the app speaks it | Source of truth |
|---|---|---|
| **Georgian (ka)** | RECORDED CLIPS only, via `AUDIO_MANIFEST`. Runtime TTS is HARD-BANNED (`speakOne` returns silent if no ka clip + no real ka voice — never reads Georgian with an en/ru voice). | `AUDIO_MANIFEST` + §6 coding rule |
| **English (en)** | RUNTIME Web Speech (`SpeechSynthesisUtterance`), device voice chosen by `pickVoice()`. | `core.js` `pickVoice` / `speakOne` |

**English runtime settings that DEFINE the app's English sound (match these exactly):**
- **Voice family (warm-preference order):** Samantha, Karen, Moira, Tessa, Google US English, Microsoft Aria, Microsoft Jenny, Fiona → fallback: any `en-*`. All are warm, female, child-friendly.
- **Rate:** `0.72` base (young child ×0.86 ≈ `0.62`). Deliberately slow and calm.
- **Pitch:** `1.1`. **Volume:** `0.85` (softened for kids, owner 2026-06-22).
- **Language tag:** `en-US`.

## 2. The root-cause CLASS (§14) — [FACT]

**Class:** *NikoLearn has no PINNED English voice identity.* Georgian is pinned (one recorded set,
identical on every device). English is **device-dependent runtime TTS** — so the exact voice varies
by phone, and ANY pre-generated English audio (a video, a marketing clip, a deliverable) will differ
from what a given device produces unless it is deliberately matched to the profile in §1.

- **Instance found (NB-89):** the „1 apple" deliverable used a different generator/voice than §1.
- **Why it matters (product, not just QA):** two children on two devices already hear two slightly
  different „Niko" English voices. Every off-profile clip widens that gap and erodes the single warm
  identity that builds parent trust. Georgian solved this by recording; English has not.

## 3. THE GATE — every non-runtime audio clip must pass ALL of these before it ships

1. **Declare the target.** State language + who hears it (kid in-app / parent / video narration).
2. **Georgian → clips only.** Never generate Georgian narration with a TTS that is not the recorded
   clip voice. If a needed ka line has no clip, it must be recorded in the SAME voice as `AUDIO_MANIFEST`,
   never synthesized ad-hoc. (This keeps §6 intact for created audio too.)
3. **English → reproduce the §1 profile.** Warm female child voice, rate ≈0.72 (0.62 for young),
   pitch ≈1.1, calm. If generated by a non-browser engine (ElevenLabs / Azure / etc.), pick the voice
   + settings that AUDIBLY match §1, not just „an English voice".
4. **A/B VERIFY (the teeth — audio's §6c step).** Play the new clip back to back with a reference clip
   from the running app (same phrase if possible). Confirm tempo, clarity, and tonality match. If a
   parent could tell „that's a different narrator", the gate FAILS → redo. Do not ship on „close enough".
   Corroborate objectively with `node tools/voice-eval.mjs` (physics: speech-rate + loudness) so „it
   feels the same" is backed by measured rate ≈ the §1 target, not just impression.
5. **Log the pass.** One line in the deliverable: voice + engine + rate/pitch used + „A/B matched to
   app reference: yes". No log = not gated = not shippable.

> Per the [no-loud-audio-test-playback] owner rule: run the A/B check muted/inspected or with the owner's
> go-ahead, never blast test audio while he is working.

## 4. OPEN PRODUCT DECISION (owner fork) — pin the English identity

The gate above makes DELIVERABLES consistent, but the underlying class (device-dependent English)
stays until the owner picks how to PIN the English voice. This is a product/architecture call:

| Option | What it means | Trade-off | 
|---|---|---|
| **A — record English clips** (like Georgian) | One recorded English voice set in `AUDIO_MANIFEST`. | Max consistency, identical on every device. More production work + app size. |
| **B — lock one TTS voice+settings as canonical** (RECOMMENDED, default) | Standardize on one named voice family + the §1 settings; match every generated clip to it; keep runtime Web Speech but document the target. | Light, no new assets. Still device-dependent at runtime where the exact voice is unavailable, but the target is defined so drift is catchable. |
| **C — status quo** | Gate deliverables only; leave runtime English as-is. | Zero work; the cross-device English gap persists. |

**Recommendation:** B now (define + document the canonical English target so nothing drifts, incl.
the video-tale voice), revisit A only if/when English becomes a core paid surface. **Default if owner
is silent:** B — treat the §1 warm profile as the pinned target for all created audio.

---
*Honor this like §6c. A visual change is not done until it is rendered and looked at; an audio clip is
not done until it is heard back-to-back with the app and matches. See [[NB-89]], PI-104.*
